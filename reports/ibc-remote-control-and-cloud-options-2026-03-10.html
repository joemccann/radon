Remote Control and Cloud Hosting Options for IBC

Research summary for operating the existing local IB Gateway / IBC service from an iPhone, and for migrating the service to cloud infrastructure with acceptable operational controls.

Decision summary 2 tracks
  • Best local path: keep IBC on the Mac and add a private remote control layer over Tailscale.
  • Best UX variant: small status/start/stop/restart endpoint served only inside the tailnet.
  • Best cloud path: dedicated Linux VM with IB Gateway + IBC, private network access only.
  • Constraint that does not go away: Sunday login / 2FA still requires human action.

Local remote-control options Recommended first
Option Fit Strengths Risks / limits
Tailscale + private web control endpoint Best overall One-tap iPhone workflow, no public exposure, reuses current wrapper scripts, easy to audit. Mac must stay powered and reachable. Requires a tiny local service to shell out safely.
Tailscale + SSH Fastest to ship No custom service required. Uses the existing scripts directly from an iPhone SSH client. Worse UX, easier to mistype commands, less status visibility.
Expose IBC command port 7462 Do not use Minimal setup. Unsafe design. Raw control port should never be exposed outside a trusted private boundary.
Remote desktop only Recovery path only Useful when Gateway is in an unexpected GUI state. Heavyweight, fragile on phone, not a clean service-control plane.

Recommended local implementation: Tailscale on Mac + iPhone, then a narrow control app that runs only status, start, stop, and restart against the existing wrappers.

Why the current setup maps well No secret rewrite
Control surface already exists
The remote layer does not need to understand IBC internals. It only needs to trigger the existing launchd wrapper scripts in ~/ibc/bin/.
Secrets stay on the Mac
The control layer never handles IB credentials directly. run-secure-ibc-gateway.sh pulls them from Keychain locally at start time.
Service semantics stay unchanged
launchd remains the lifecycle manager. You are adding remote invocation, not replacing the current service model.
Recovery path remains available
If IBC or Gateway reaches a bad GUI state, Tailscale can still provide SSH or remote-desktop access for operator recovery.
Recommended local architecture Private control plane
Layer 1
Tailnet transport. Install Tailscale on the Mac and iPhone. Use tailnet ACLs or device identity so only your user can reach the endpoint.
Layer 2
Minimal controller. Run a tiny local service that exposes only four actions: service status, start, stop, restart.
Layer 3
Safe execution policy. The controller should dispatch only the known wrapper scripts and return exit status plus current launchd state. No arbitrary shell input.
Layer 4
Operator observability. Include last action timestamp, current launchd state, and a small tail of ~/ibc/logs/ibc-gateway-service.log.
Cloud hosting options Migration track
Option Recommendation Operational profile Main downside
Self-managed Linux VM Best cloud path Dedicated VM, native IB Gateway install, native IBC, Tailscale/private ingress, persistent state volumes. Still outside IBKR's supported headless model. You own the weekly re-auth and operational drift.
QuantRocket Good if broader stack needed More complete IB operating environment with documentation around deployment and multiple gateways. Heavier and more opinionated than needed if the requirement is only Gateway uptime and control.
Community Docker images Use only if ops-ready Reproducible deployment, easier rebuilds, container-native workflows. Leans most heavily on unsupported headless/containerized patterns. Easy to misconfigure networking or persistence.

Cloud recommendation order: Linux VM first, QuantRocket second, Docker only if you already run secure private container infrastructure well.

Cloud constraints Hard limits
  • IBKR API requires TWS or IB Gateway. There is no standalone API daemon.
  • Headless auto-launch is not officially supported by IBKR. Cloud operation is viable but outside the strict support boundary.
  • Sunday re-auth persists. Daily unattended restarts are possible, but weekly re-auth still requires operator action.
Security posture Private-only
  • Never expose IB API, VNC, or IBC control ports publicly. Use VPN or tailnet-only access.
  • Persist settings and state. Gateway API settings and session-recovery files must survive restarts.
  • Constrain control paths. Remote operators should access service wrappers, not a generic shell by default.
Session behavior Operator awareness
  • Single-user conflicts matter. Other IBKR session activity can interrupt a Gateway session.
  • Market data entitlements follow usernames. Separate concurrent apps may require multiple usernames.
  • Plan for human recovery. Keep SSH or remote desktop available for unusual GUI or session states.
Implementation plan Phased rollout
Phase 1
Fastest safe version. Enable Tailscale on Mac and iPhone. Validate that you can SSH in from the phone and run the wrapper scripts manually.
Phase 2
Operator UX. Build a small local service and status page that wraps the four known actions and returns service/log health.
Phase 3
Availability hardening. Decide whether the Mac should stay awake when unattended or whether you need a dedicated always-on node.
Phase 4
Cloud pilot. Stand up a private Linux VM, install Gateway + IBC, test reboots and session recovery for at least one full weekly cycle before cutover.
Recommendation order Final ranking
  1. Deploy a private Tailscale-based control plane on the existing Mac. Lowest risk, least invasive, reuses the current launchd/Keychain model.
  2. If cloud uptime is required, migrate to a dedicated Linux VM. Keep the VM private and accept the unsupported-headless tradeoff.
  3. Use QuantRocket only if you want a fuller operating platform. Good productized option, but broader than the stated need.
  4. Use Docker only if you already manage secure container operations well. Do not make containerization the first move.
